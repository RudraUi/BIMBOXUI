PLANNING SECTION DOCUMENTATION
Layman Understanding Guide

File purpose:
This document explains the Planning section in simple language. It covers all main views, buttons, fields, schedule parameters, calculations, and how the user is expected to use them.

Current scope:
The Planning section is a frontend/local planning workspace. It does not use a backend database yet. Any data created or edited is stored only in the running app state unless backend/local persistence is added later.


1. WHAT THE PLANNING SECTION IS

The Planning section is used to manage construction schedule work.

It helps the team answer:
- What work is planned?
- What phase does the work belong to?
- When should it start and finish?
- What is actually happening on site?
- Which work is late?
- Which work is ready for the next few weeks?
- What is the approved baseline schedule?
- What milestones are important?
- What dependencies control one activity after another?

The Planning section is organized like this:
- Phase / Work Package
- Task
- Subtask

Example:
Phase: Site Mobilization
Task: Temporary Power, Water & Safety Setup
Subtask: Temporary power connection


2. MAIN DATA STRUCTURE

2.1 Phase / Work Package

A phase is a big section of work.

Examples:
- Pre-Construction
- Site Mobilization
- Substructure
- Superstructure
- MEP Rough-in

Fields used in a phase:

ID:
Internal technical ID used by the app.

Code:
Numbering like 1, 2, 3. It helps create WBS-style hierarchy.

Name:
Phase name shown to the user.

Owner:
Main department responsible for the phase.

Status:
Overall phase condition, calculated or selected from tasks.

Progress:
Average progress of all tasks inside the phase.

Start:
Earliest task start date inside the phase.

End:
Latest task finish date inside the phase.

Tasks:
List of tasks inside that phase.


2.2 Task

A task is the main schedule activity under a phase.

Example:
Task: Design Freeze & Consultant Sign-off

Fields used in a task:

ID:
Internal technical ID.

Code:
Numbering like 1.1, 1.2, 2.1.

Activity ID:
Construction schedule activity number shown to the user. If not manually set, it uses the code.

Name:
Task name.

Owner / Responsible:
Department or team responsible for the task.

Status:
Current schedule condition.

Progress:
Actual progress percentage from 0 to 100.

Start:
Planned start date.

End:
Planned finish date.

Actual Start:
Date when work actually started on site.

Actual Finish:
Date when work actually finished on site.

Actual Days:
Actual duration. If Actual Start and Actual Finish are entered, the app calculates it automatically.

Float:
Extra available time before the task affects another activity. It is read-only and calculated from schedule logic.

Constraint:
Rule that limits how the task can move in the schedule.

Dependencies:
Predecessor activities that control when this task can happen.

Buffer:
Planned extra time added intentionally by planners.

Delay:
Unplanned extra time caused by execution problems.

Remarks:
Short note or site update.

Recovery Plan:
What the team will do to recover delay.

Color:
Timeline / Gantt bar color.

Subtasks:
Smaller steps under the task.


2.3 Subtask

A subtask is a smaller work item under a task.

Example:
Task: Temporary Power, Water & Safety Setup
Subtask: Temporary power connection

Subtasks use almost the same fields as tasks:
- Activity ID
- Name
- Responsible
- Status
- Progress
- Start
- Finish
- Actual Start
- Actual Finish
- Actual Days
- Float
- Constraint
- Dependency
- Buffer
- Delay
- Remarks
- Recovery Plan
- Color

Difference:
A subtask always belongs to a parent task.

40. MODAL SECTION: DEPENDENCY MANAGEMENT

Dependency means one activity depends on another.

Example:
Concrete pouring cannot start before reinforcement is complete.

Each dependency row has:

Predecessor Activity:
The activity that controls the current activity.

Dependency Type:
Relationship between predecessor and current activity.

Lag:
Extra gap or overlap in days.

Remove button:
Deletes dependency row.

Add Dependency:
Adds a new dependency row.


41. DEPENDENCY TYPES

FS - Finish to Start:
Current activity starts after predecessor finishes.

Example:
Painting starts after plaster finishes.

SS - Start to Start:
Current activity starts after predecessor starts.

Example:
Inspection starts after installation starts.

FF - Finish to Finish:
Current activity finishes after predecessor finishes.

Example:
Testing finishes only after installation finishes.

SF - Start to Finish:
Current activity finishes after predecessor starts.

Less common, but supported.


42. LAG INPUT

Lag means extra days added or subtracted from dependency logic.

Examples:

+2d:
Wait 2 days after predecessor logic.

-1d:
Start/finish 1 day earlier than dependency point.

+5d:
Add 5 days gap.

45. BUFFER VS DELAY

Buffer:
Planned safe time.
Added intentionally before execution.
Used to absorb uncertainty.
Shown as faded/dashed bar.
Should not be treated as failure.

Delay:
Unplanned slippage.
Happens because actual work did not follow plan.
Can affect successor, milestone, or critical path.
Needs reason and recovery plan.

Example:
Planned Finish: 10 June
Buffer: 3 days
Actual Finish: 12 June

Delay from planned finish = 2 days.
Buffer consumed = 2 days.
Remaining buffer = 1 day.
Net delay = 0 days.
This should not be marked as critical delay.


46. AUTOMATIC CALCULATIONS

Progress:
If status is Completed, progress becomes 100%.
If actual start and actual finish exist, progress can be calculated from actual days vs planned duration.
If actual dates are incomplete, entered progress is preserved.

Actual Days:
Actual Finish - Actual Start.

Delay Days:
If completed: Actual Finish - Planned Finish.
If not completed and planned finish passed: Today - Planned Finish.

Planned Progress:
Based on today's date between planned start and planned finish.

Variance:
Actual Progress - Planned Progress.

Float:
Calculated from dependency/successor schedule logic where possible.

Phase Progress:
Average progress of tasks inside the phase.

Phase Start:
Earliest task start.

Phase Finish:
Latest task finish.

Phase Status:
Calculated from task statuses.


47. CRITICAL PATH LOGIC

Current critical path logic is simplified.

An item is treated as critical if:
- Float is zero or negative
- Delay impact says Impacts Critical Path
- Delay days exceed buffer plus a small threshold

Important:
This is not a full Primavera/MS Project CPM engine yet.

50. CURRENT FRONTEND-ONLY LIMITATIONS

No backend persistence:
Data can be lost on refresh.

No user login:
Responsible/assignee values are simple labels.

No full CPM scheduling engine:
Dependency and float logic is simplified.

No import/export to Primavera/MS Project yet:
Only CSV export is available.

No permanent audit trail:
Baseline and updates are local state only.

No real document/photo storage:
Uploaded photo stores file name only in local state.
